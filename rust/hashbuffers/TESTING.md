# Testing hashbuffers

## Unit tests

```bash
cargo test
```

## Fuzzing

Uses [cargo-fuzz](https://rust-fuzz.github.io/book/cargo-fuzz.html) with libFuzzer.
Requires nightly: `rustup install nightly`.

### Targets

| Target | What it does |
|---|---|
| `fuzz_block` | Parses input as all four block types, exercises all accessors |
| `fuzz_table` | TABLE-specific: vtable accessors, direct data, sub-blocks, links |
| `fuzz_data` | DATA-specific: iteration, typed slice casting |
| `fuzz_slots` | SLOTS-specific: element access, iteration |
| `fuzz_links` | LINKS-specific: link lookup at various indices |

### Running

```bash
# List available targets:
cargo fuzz list

# Run a target (runs until crash or Ctrl-C):
cargo fuzz run fuzz_block

# Time-limited run:
cargo fuzz run fuzz_block -- -max_total_time=3600

# Parallel workers (collects multiple crashes, respawns on crash):
cargo fuzz run fuzz_block -- -fork=2 -max_total_time=3600

# Reproduce a crash artifact:
cargo fuzz run fuzz_block fuzz/artifacts/fuzz_block/crash-<hash>
```

### Seed corpus

Hand-crafted seeds live in `fuzz/corpus/{block,table,data,slots,links}/` and are
generated by `fuzz/generate_seeds.rs`. Regenerate with:

```bash
cd fuzz && rustc generate_seeds.rs && ./generate_seeds
```

The fuzzer also builds up auto-discovered corpus files in `fuzz/corpus/fuzz_*/`
during runs. These are gitignored but preserved locally between runs, so the
fuzzer picks up where it left off.

### Workflow

The fuzzer **stops on the first crash** and saves the input to
`fuzz/artifacts/<target>/`. After a crash:

1. Examine the artifact and fix the bug
2. Re-run on the artifact to confirm: `cargo fuzz run <target> fuzz/artifacts/<target>/crash-<hash>`
3. Resume fuzzing

Don't leave the fuzzer running unattended if you expect crashes — it will
rediscover the same bug repeatedly and waste CPU. Do a shorter run first, fix
everything, then do the long run for deeper coverage.

### What's in git

- `fuzz/Cargo.toml`, `fuzz/Cargo.lock` — build config
- `fuzz/fuzz_targets/*.rs` — the targets
- `fuzz/generate_seeds.rs` — seed generation script
- `fuzz/corpus/{block,table,data,slots,links}/` — hand-crafted seeds

Gitignored (see `fuzz/.gitignore`):
- `fuzz/target/` — build artifacts
- `fuzz/corpus/fuzz_*/` — auto-discovered corpus
- `fuzz/artifacts/` — crash files

## Miri

[Miri](https://github.com/rust-lang/miri) checks for undefined behavior:
uninitialized reads, invalid pointer provenance, aliasing violations, etc.

### Unit tests under Miri

```bash
cargo +nightly miri test
```

### Corpus replay under Miri

Replays all fuzzer corpus files through the same exercisers as the fuzz targets,
under Miri's UB checker. This combines the fuzzer's coverage-guided input
discovery with Miri's deep semantic checks.

```bash
MIRIFLAGS="-Zmiri-disable-isolation" cargo +nightly miri test --test miri_corpus
```

`-Zmiri-disable-isolation` is required because the test reads corpus files from
disk, which Miri's default sandbox blocks.

This is slow (~1000x native speed) so it's not practical to run Miri as the
fuzzer itself. Instead: fuzz first to build a good corpus, then replay that
corpus under Miri.
